Deployment

Docker Compose for development, Kubernetes for production.

Docker Compose (Development)

The quickest way to get TokenSentinel running locally:

docker compose -f proxyops_gateway/docker-compose.yml up -d

This starts all 6 services:

Verification

After starting, verify all services are healthy:

# Health check
curl http://localhost:3001/api/admin/health

# Dashboard HTML
curl http://localhost:3001/

# Proxy endpoint (requires auth)
curl -X POST http://localhost:3000/v1/chat/completions \
  -H "Authorization: Bearer dev-key-123" \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o-mini","messages":[{"role":"user","content":"hi"}]}'

Stopping

docker compose -f proxyops_gateway/docker-compose.yml down -v

The -v flag removes persisted data volumes (PostgreSQL data and file uploads). Omit it to preserve data between restarts.

Production Considerations

  • TLS termination: The rust-proxy handles TLS. Provide a certificate via volume mount or use an ingress controller with TLS.
  • PostgreSQL: For production, use a managed PostgreSQL service (RDS, Cloud SQL, etc.) instead of the Docker container.
  • Redis: Use a managed Redis service or configure Redis Sentinel/Cluster for high availability.
  • Horizontal scaling: rust-proxy and go-router are stateless and can be scaled horizontally behind a load balancer.
  • Secrets: Store API keys in Kubernetes Secrets or your cloud provider's secret manager.

E2E Validation

After deployment, run the E2E validation script to verify everything works:

Linux / macOS

./deploy/run-e2e.sh

Windows PowerShell

.\deploy\run-e2e.ps1

The script verifies: health endpoints, cost recording, dashboard access, SSE events, and data persistence.

E2E Validation Script

The E2E validation script (deploy/run-e2e.sh / deploy/run-e2e.ps1) performs these checks:

  1. Health check: GET /api/admin/health returns {"status": "ok"}
  2. Cost recording: Creates a test cost entry and verifies it appears in the cost summary
  3. Dashboard HTML: Fetches GET / and checks for expected HTML content
  4. SSE events: Opens the SSE stream and verifies events are received
  5. Data persistence: Verifies cost data is stored in PostgreSQL

Troubleshooting

Services fail to start

Port conflicts Ensure ports 3000, 3001, 6379, 8080, and 5432 are not already in use.

PostgreSQL connection refused

PostgreSQL takes a few seconds to initialize on first start. The cost-dashboard waits with a health check, but if it still fails, check the postgres logs:

docker compose -f proxyops_gateway/docker-compose.yml logs postgres

Redis connection errors

Verify Redis is running and accessible:

docker compose -f proxyops_gateway/docker-compose.yml exec redis redis-cli ping
# Expected: PONG

No data in dashboard

Ensure requests are routed through the gateway. Direct requests to upstream providers bypass cost recording. Verify the go-router is processing requests by checking its logs:

docker compose -f proxyops_gateway/docker-compose.yml logs go-router
