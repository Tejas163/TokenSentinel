Architecture

How TokenSentinel routes, governs, and observes every LLM request.

System Overview

TokenSentinel is a distributed gateway that sits between LLM clients and upstream AI providers. Every request passes through a chain of services that handle authentication, rate limiting, routing, cost tracking, and observability.

Request Flow

  1. Client sends an LLM request to rust-proxy:3000 with an API key.
  2. Rust authenticates the key, checks the Redis rate-limit counter (ratelimit:{key}, 60s TTL), validates payload size (≤10MB), strips sensitive headers (Authorization, Cookie, etc.).
  3. Rust forwards the request to go-router:8080 with augmented headers (X-Request-ID, X-Authenticated-User, X-Rate-Remaining).
  4. Go resolves the route pattern from Redis (routes:{pattern}), selects an upstream provider using weighted random selection (or autonomous model tier), checks the team budget via X-Team-Name.
  5. Go proxies the request to the selected upstream AI provider with retry/backoff. On success, adds X-Model-Used and X-Cost-Cents response headers.
  6. Go writes cost telemetry to Redis (sentinel:{request_id}:cost, 24h TTL) and publishes to health:events pub/sub.
  7. Erlang monitor picks up events via pub/sub, updates heartbeats.
  8. Cost dashboard subscribes to events, reads cost data, inserts into PostgreSQL. Browser SSE connections stream live cost + anomaly events.
