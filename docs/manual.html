User Manual

Complete guide to using TokenSentinel — from daily operations to advanced configuration.

Cost Dashboard

The cost dashboard is the primary interface for monitoring AI spend. It is accessible at http://localhost:3001.

Summary Cards

The dashboard header displays six summary metrics for the selected period:

  • Requests — number of proxied requests
  • Total Tokens — sum of input + output tokens
  • Input Tokens — prompt tokens consumed
  • Output Tokens — completion tokens generated
  • Models Used — count of distinct models in the period
  • Avg Tokens/Req — average token consumption per request

Time Period Selection

Toggle between predefined periods: 1H, 6H, 24H, 3D, 7D. The dashboard reloads data on selection change.

Charts

The dashboard includes two interactive charts powered by Chart.js:

  • Cost Over Time — stacked line chart showing cost per model across hourly buckets. Hover to see exact dollar amounts per model at each time slice.
  • Cost by Model — doughnut chart showing the proportional cost distribution across models. Hover to see the percentage and dollar amount for each model.

Both charts refresh automatically when the period changes, when new SSE cost events arrive, or after seeding demo data.

Cost by Model Table

A detailed table shows cost broken down per model, with columns for requests, input tokens, output tokens, total tokens, avg input, and avg output.

Live Events (SSE)

When the page is open, a green badge indicates an active SSE connection. New cost events and anomaly alerts stream in real time without page refreshes. A red badge indicates a disconnected state with automatic reconnection.

Onboarding Wizard

On first load with no data, the dashboard shows a guided onboarding wizard with three steps:

  • Seed Demo Data — populate the dashboard with 7 days of realistic cost data across 7 models and 4 teams
  • Explore the Dashboard — use the period selector and watch for SSE events
  • Run an Assessment — navigate to the prescriptive engine for cost-saving recommendations

Anomaly Detection

TokenSentinel detects cost anomalies using a 3-sigma sliding window algorithm:

  • Computed per-model over recent time windows
  • Checked every 5 minutes (configurable via ANOMALY_INTERVAL)
  • Anomalies are broadcast via SSE to the dashboard
  • Available via REST: GET /api/dashboard/anomalies?limit=10
Pro Tip Keep the dashboard open on a monitor for real-time AI cost observability during development.

Error Response Format

All error responses use a structured JSON format:

{
  "error": "input of 50000 tokens exceeds model gpt-4 limit of 8192 tokens",
  "request_id": "gen-1717718400000000000",
  "error_code": "rate_limited"
}

request_id is always included when available. error_code is included for rate limiting ("rate_limited") and other machine-classifiable errors.

Rate Limiting

The go-router includes a per-client rate limiter using an in-memory token bucket:

  • Clients are identified by X-Api-Key header, falling back to RemoteAddr
  • Default capacity: 60 tokens (max burst)
  • Default refill rate: 60 tokens/second
  • Exceeding the limit returns 429 Too Many Requests with Retry-After: 1 header
  • Configuration: RATE_LIMIT_CAPACITY and RATE_LIMIT_REFILL env vars

Note Token estimation uses the tiktoken library for OpenAI models and character-based estimation for others. These are estimates, not exact counts from the upstream provider.

Semantic Caching

TokenSentinel includes a semantic caching system that avoids re-prompting identical or semantically similar requests. Two backend modes are available:

MinHash LSH (default)

A pure-Go implementation that works with any Redis instance (redis:7-alpine):

  • Tokenizes prompts into trigrams
  • Computes a 64-hash FNV-1a MinHash signature (~33µs per prompt)
  • Uses 16×4 LSH banding for approximate nearest neighbor search
  • Stores signatures in Redis with configurable TTL
  • Similarity threshold: configurable via SEMANTIC_CACHE_THRESHOLD (default 0.85)

Vector Embeddings

Uses an embedding API (OpenAI-compatible) for higher accuracy on rephrased prompts:

  • Calls a configurable embedding API (EMBEDDING_API_URL)
  • Stores vectors via RediSearch FT.CREATE + FT.SEARCH with COSINE distance
  • Caches embeddings in Redis (24h TTL)
  • Requires redis-stack-server (includes RediSearch module)
  • Configuration: EMBEDDING_API_URL, EMBEDDING_MODEL, EMBEDDING_DIM
Benchmark MinHash: 33µs per signature, catches exact+shallow rephrases at 0.85+ threshold. Embedding: 1.09ms per call (including API latency), catches 3/5 rephrased variants. For best results in production, use embedding mode with a local model (e.g., LM Studio on localhost).

Cache Headers

When semantic cache is enabled, responses include cache headers:

  • X-Cache: HIT — response was served from cache (zero upstream cost)
  • X-Cache: MISS — response was fetched from upstream
  • X-Cache-Savings-Cents — the dollar amount saved by the cache hit

Routing Configuration

Routes are stored in Redis as JSON under routes:{path}. Each route defines providers with weights, models, and timeouts:

{
  "pattern": "/v1/chat/completions",
  "providers": [
    {
      "url": "https://api.openai.com/v1/chat/completions",
      "model": "gpt-4o",
      "weight": 3,
      "timeout": 30
    },
    {
      "url": "https://api.anthropic.com/v1/messages",
      "model": "claude-3-sonnet",
      "weight": 1,
      "timeout": 30
    }
  ],
  "auto_model": false
}

Setting a Route

docker compose exec redis redis-cli SET "routes:/v1/chat/completions" \
  '{"pattern":"/v1/chat/completions","providers":[{"url":"https://api.openai.com/v1/chat/completions","model":"gpt-4o","weight":3,"timeout":30}],"auto_model":false}'

Adaptive Routing

TokenSentinel dynamically adjusts provider selection weights based on historical success and error rates:

  • Each provider's error rate is tracked via recordProviderSuccess and recordProviderFailure
  • Dynamic weight = baseWeight × (1 − errorRate), with a floor of 5% of the original weight
  • Weights are recomputed on every selection
  • At debug log level, weight changes are logged (e.g., gpt-4@url: 3→2 (err 33.3%))

This means a provider that starts failing gets gradually deprioritized. The weight floor ensures it still gets occasional test traffic so it can recover.

Fallback Routing

When the primary provider fails, TokenSentinel uses a multi-provider fallback chain:

  1. Primary target — the originally selected provider
  2. Equivalence group — providers serving equivalent models (e.g., gpt-4 fallbacks to claude-3-sonnet, gpt-4o-mini, gpt-3.5-turbo)
  3. Cheapest remaining — all remaining providers sorted by cost

Providers with open circuit breakers are skipped. When a fallback provider handles the request, the response includes X-Cost-Routing: fallback. The model equivalence map is defined in fallback_router.go and covers all major providers.

Circuit Breaker

Each upstream provider has a circuit breaker that opens after 5 consecutive failures, with a 30-second cooldown before attempting half-open probes. This prevents cascading failures to unhealthy providers.

Per-Team Budgets

TokenSentinel supports per-team budget management for cost allocation across departments or projects.

Creating a Team

curl -X POST http://localhost:3001/api/admin/teams \
  -H "Authorization: Bearer $AUTH_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"name": "engineering", "monthly_token_limit": 10000000}'

Budget-Aware Routing

When a team exceeds its budget, the gateway automatically reroutes to the cheapest available provider. This is triggered via the X-Team-Name request header:

curl -X POST http://localhost:3000/v1/chat/completions \
  -H "Authorization: Bearer dev-key-123" \
  -H "X-Team-Name: engineering" \
  -H "Content-Type: application/json" \
  -d '{"model": "gpt-4o", "messages": [...]}'

Budget Alert Webhooks

Configure webhooks to notify Slack or Teams when a team approaches its budget limit:

curl -X POST http://localhost:3001/api/admin/budget-rules \
  -H "Authorization: Bearer $AUTH_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "team_name": "engineering",
    "threshold_pct": 80,
    "webhook_url": "https://hooks.slack.com/services/..."
  }'

Webhooks fire when usage crosses the threshold percentage. A 30-minute cooldown prevents duplicate notifications.

Slack/Team Cost Digests

Configure scheduled cost digests sent to Slack or Teams webhooks:

VariableDescription
DIGEST_WEBHOOK_URLSlack or Teams incoming webhook URL

Digests include a per-model cost breakdown formatted as Slack message attachments.

Monitoring & Alerts

The cost dashboard includes a background monitoring engine that tracks spend trends and generates alerts:

  • Spend trend monitoring — detects spikes or drops per model (every MONITORING_INTERVAL, default 5m)
  • Anomaly detection — 3-sigma z-score vs trailing window (every ANOMALY_INTERVAL, default 5m)
  • Savings tracking — records cost savings events from semantic cache hits
  • Alerting — email via SMTP, webhook (Slack/Teams), and SSE events to the dashboard
  • Data retention — automatic pruning after RETENTION_MAX_DAYS (default 90)

Alert Severity Levels

Alerts are classified into three severities:

  • critical — red, urgent attention required (e.g., spend spike > 5x)
  • warning — yellow, moderate concern (e.g., spend spike > 2x)
  • info — blue, informational (e.g., savings milestone reached)

Authentication

TokenSentinel uses a shared API key for service-to-service authentication:

  • Client → Go Router: X-Api-Key header or Authorization: Bearer <key>
  • Dashboard/admin API: Same X-Api-Key or Authorization: Bearer pattern
  • Dashboard HTML page: data-api-key attribute on <body>, used for AJAX calls
  • Rust Proxy → Go Router: Forwards the API key from the client

Set the shared key via AUTH_API_KEY environment variable. When empty, auth is disabled in development.

API Playground

The dashboard includes a built-in API playground for testing cost ingestion without real LLM traffic:

  • Select a model, enter input/output token counts, and see estimated cost
  • Send simulated requests that flow through the same cost pipeline as production traffic
  • Results appear in the dashboard in real time via SSE

Prescriptive Engine

TokenSentinel includes a prescriptive cost optimization engine accessible via the Assessments page (/assessments):

  • Cost Assessment — full analysis with model substitution, GPU tier, and provider recommendations
  • What-If Simulation — adjust volume, model mix, or provider to see cost impact
  • Multi-Scenario Comparison — compare up to 5 scenarios side-by-side
  • Reports — exportable as HTML, CSV, or PDF
  • Starter Templates — pre-configured templates for startup, mid-size, and enterprise deployments

MCP Gateway

The MCP gateway (port 3010) exposes 11 cost-management tools via the Model Context Protocol:

  • Cost queries — get_cost_summary, get_model_costs, get_anomalies
  • Assessments — run_assessment, run_whatif, whatif_multi_scenario, whatif_volume_shift, whatif_model_switch
  • Budgets — get_budget_status, list_budget_rules
  • Reports — get_report

MCP uses SSE (GET /mcp) for streaming sessions and JSON-RPC (POST /mcp) for tool calls. Team data scoping via AGENT_TEAM_MAP ensures users only see their team's costs. Budget enforcement blocks tool calls for teams that have exceeded their budget.

Data Retention

Cost history entries are automatically pruned after 90 days (configurable via RETENTION_MAX_DAYS). The pruning runs periodically on the cost dashboard service.
